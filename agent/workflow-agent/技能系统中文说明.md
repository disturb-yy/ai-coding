# 技能系统中文说明

本文件是一份临时中文说明，独立存在，不作为系统配置、技能定义或运行时规范的一部分。

## 目标

当前实现的是一套面向 workflow-agent 的 Skill Execution System。它把每个技能定义为确定性的执行单元：接收标准输入和配置，读取最小必要上下文，输出结构化 artifact。

技能不是 Agent，不负责控制 workflow，也不是自由提示词。workflow 的阶段流转、skill 选择、artifact 持久化和最终总结仍然只由 workflow-agent 控制。

## 目录

核心目录包括：

- `workflow/WORKFLOW.md`：全局 workflow 单一事实源。
- `AGENT.md`：Codex/OpenCode 通用 orchestrator 说明。
- `codex/AGENTS.md`：Codex 适配层。
- `opencode/agents/workflow-agent.md`：OpenCode 唯一 agent 入口模板。
- `opencode/workflow-agent/`：OpenCode 外部支持文件模板。
- `runtime/skill-executor.md`：skill 执行器规范。
- `runtime/skill-registry.md`：skill 注册表规范。
- `config/workflow-skills.json`：当前启用的 workflow skill 映射。
- `config/workflow-skills.example.json`：skill 映射模板。

核心技能目录包括：

- `skills/requirement-analysis/`
- `skills/project-understanding/`
- `skills/solution-design/`
- `skills/code-implementation/`
- `skills/ut-generation/`
- `skills/workflow-skill-gen/`

每个 workflow-compatible skill 通常包含：

- `SKILL.md`：技能职责、输入、输出、禁止行为、执行规则。
- `config.schema.json`：严格 JSON 配置 schema。
- `example.json`：调用示例。
- `agents/openai.yaml`：可选，用于 Codex UI metadata。
- `references/`：可选，用于按需加载的参考资料。

## OpenCode 安装架构

OpenCode 使用“唯一 agent 文件 + 外部支持文件”的架构。

实际安装路径：

```text
~/.config/opencode/agents/
  workflow-agent.md

~/.config/opencode/workflow-agent/
  config/
  runtime/
  skills/
  workflow/
```

含义：

- `~/.config/opencode/agents/workflow-agent.md` 是唯一会被 OpenCode 扫描成 agent 的入口文件。
- `~/.config/opencode/workflow-agent/` 是支持目录，不作为 agent 扫描。
- `config/` 放 `workflow-skills.json`。
- `runtime/` 放 `skill-executor.md` 和 `skill-registry.md`。
- `skills/` 放 workflow-compatible skills。
- `workflow/` 放 `WORKFLOW.md` 等 workflow 核心规范。

OpenCode agent 入口只做 orchestration，不承载完整 workflow 逻辑。它通过相对路径读取：

```text
../workflow-agent/workflow/WORKFLOW.md
../workflow-agent/config/workflow-skills.json
../workflow-agent/runtime/skill-executor.md
../workflow-agent/runtime/skill-registry.md
```

这样可以避免 OpenCode 把 runtime、skills、workflow 支持文件误识别为多个 agent。

## 执行模型

每个技能按固定流程执行：

1. 校验输入。
2. 根据 `config/workflow-skills.json` 解析 skill registry entry。
3. 加载目标 `SKILL.md`。
4. 校验 `config.schema.json`。
5. 准备最小上下文。
6. 解析执行模式：`direct` 或 `delegated`。
7. 执行技能逻辑。
8. 返回结构化 artifact。

Codex 默认使用 `direct` 顺序执行。OpenCode 可以在必要时使用 `delegated`，把 skill 交给 subAgent 执行，但 subAgent 不能决定 workflow 阶段流转。

## Skill 如何配置进去

使用 `workflow-skill-gen` 改造现有 skill 后，需要把生成的 registry entry 加入：

```text
config/workflow-skills.json
```

如果是给 OpenCode 实际使用，配置位置是：

```text
~/.config/opencode/workflow-agent/config/workflow-skills.json
```

如果是在当前仓库中维护 OpenCode 模板，配置位置是：

```text
opencode/workflow-agent/config/workflow-skills.json
```

coding skill 示例：

```json
{
  "phase": "Implementation",
  "artifact_type": "implementation_result",
  "skill": "my-coding-skill",
  "path": "skills/my-coding-skill",
  "enabled": true,
  "priority": 100,
  "execution": {
    "codex": "direct",
    "opencode": "delegated"
  },
  "config": {
    "mode": "implement"
  }
}
```

UT skill 示例：

```json
{
  "phase": "Testing",
  "artifact_type": "test_result",
  "skill": "my-ut-skill",
  "path": "skills/my-ut-skill",
  "enabled": true,
  "priority": 100,
  "execution": {
    "codex": "direct",
    "opencode": "delegated"
  },
  "config": {
    "mode": "generate",
    "run_tests": true
  }
}
```

注册规则：

- `phase` 必须匹配 workflow 阶段。
- `artifact_type` 必须匹配 skill 输出。
- `path` 必须指向 skill 目录。
- `config` 必须能通过该 skill 的 `config.schema.json` 校验。
- 多个 skill 匹配同一阶段时，优先使用 `priority` 更高的启用项。

## Artifact 存储

每次 workflow 执行都会使用一个 `workflow_id`，并把阶段 artifact 保存到：

```text
.agent/workflow-artifacts/<workflow_id>/
```

默认文件结构：

```text
00-metadata.json
01-requirement-analysis.json
02-project-understanding.json
03-solution-design.json
04-implementation-result.json
05-test-result.json
06-verification-result.json
07-workflow-summary.json
```

如果进入 fix loop，重试 artifact 使用 attempt 后缀：

```text
04-implementation-result.attempt-2.json
05-test-result.attempt-2.json
06-verification-result.attempt-2.json
```

artifact 是 workflow 的主要状态机制。不要依赖聊天历史或隐藏记忆来传递阶段信息。

## 日志规则

workflow 日志保存到：

```text
.agent/workflow-logs/<workflow_id>.log
```

日志采用 kv 格式，方便 grep 和结构化分析：

```text
[workflow-agent] event=phase_start phase=Implementation prev=SolutionDesign confidence=0.82
```

必须记录：

- workflow start / complete / failed
- phase start
- phase skip
- skill start
- skill return
- subAgent spawn / return
- subAgent spawn_failed
- fallback_local
- confidence 变化
- retry 和 root cause
- artifact 写入路径

跳过阶段时必须写明 `skip_reason`。每个阶段都要记录 confidence，方便后续判断信息增益和执行质量。

## 最终总结

总体阶段执行完成后，必须输出 `workflow_summary` artifact。

summary 至少包含：

- 是否完成。
- 修改文件。
- 验证证据。
- 每个阶段的状态。
- 每个阶段使用的 skill。
- 每个阶段 artifact 路径。
- 每个阶段 confidence。
- 跳过阶段的原因。
- retry 次数。
- 风险和后续事项。
- artifact/log 存储位置。

这样即使不逐个打开阶段 artifact，也能快速判断 workflow 是否可靠、哪里被跳过、哪个阶段置信度低、是否发生过 fallback 或 retry。

## 核心约束

- 技能之间不能互相调用。
- 技能不能修改 workflow 状态。
- 技能不能决定阶段流转。
- 技能不能持久化隐藏记忆。
- 技能只能通过 artifact 传递状态。
- 技能只能读取当前任务所需的最小上下文。
- Codemap 和 Understand 优先于源码读取。
- OpenCode subAgent 可以执行 skill，但不能控制 workflow。

## workflow-skill-gen

`skills/workflow-skill-gen/` 用于改造你已有的 skill，例如 coding skill、UT 生成 skill、review skill。

它会帮助生成：

- workflow-compatible `SKILL.md`
- `config.schema.json`
- `example.json`
- registry entry
- artifact 输出契约
- direct/delegated 兼容规则

改造完成后，必须把 registry entry 配置进 `config/workflow-skills.json`。如果目标是 OpenCode，则同步配置到 `~/.config/opencode/workflow-agent/config/workflow-skills.json`，否则 workflow-agent 不会自动使用该 skill。
