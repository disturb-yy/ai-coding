# OODA 索引

使用本技能生成准确、可追溯、便于后续智能体使用的索引文档。索引必须说明当前存在什么、如何组织、暴露了哪些契约，以及使用或修改这些内容时必须遵守哪些规则。

## 输出契约

对于实质性任务，可见回复必须按以下顺序组织：

```xml
<observe>
</observe>
<orient>
</orient>
<decide>
</decide>
<draft>
</draft>
<precheck>
</precheck>
<act>
</act>
<evaluate>
</evaluate>
```

每个部分保持简洁。工具调用发生在可见 XML 标签之外，但总结和决策必须遵循该顺序。

## 工作流

### `<observe>` 源材料侦察

在写入索引前，先识别用户请求的索引目标，并扫描相关项目上下文：

- 已有索引文件、README、架构文档、编码规范、路由/API 文档、schema，以及模块级注释。
- 目标范围附近的目录结构和命名模式。
- 定义公共模块、导出函数、类、类型、配置、命令、端点、事件或数据契约的源码文件。

当预期材料缺失时，要明确说明。不要虚构源码或文档中不可见的模块、签名、规则或关系。

### `<orient>` 索引标准选择

选择最适合目标的索引形态：

- **项目索引**：汇总层级、主要目录、入口点、构建/测试命令、职责边界和跨层依赖规则。
- **模块索引**：汇总模块用途、公共文件、导出项、内部依赖、扩展点和使用注意事项。
- **API 索引**：汇总可调用 API、签名、参数、返回值、副作用、错误、幂等性、认证/配置要求；只有在源码支持时才给出示例。
- **基础设施索引**：汇总日志、HTTP、缓存、时间、配置、认证、持久化、重试、序列化、队列、存储等可复用基础设施能力。优先使用已文档化的基础设施，而不是手写工具。
- **业务能力索引**：汇总业务链路、实体、流程、不变量、策略规则、外部集成和校验要求。

如果用户提供了现有本地索引约定，应优先保持该约定，除非它与准确性或用户要求的 OODA 标准冲突。

### `<decide>` 索引计划

列出目标文件、将使用的源材料、所选索引类型和章节大纲。计划必须限制在用户请求的索引范围内。包含写入后的验证命令或静态检查方式。

### `<draft>` 索引草稿

在编辑文件前，先起草完整索引内容。草稿应足够具体，便于检查：

- 范围和用途。
- 目录或模块地图。
- 可验证的公共契约，以及可获得时的精确名称和签名。
- 输入、输出、返回约定、错误、副作用和配置。
- 依赖规则，以及“使用该能力而不是手写该能力”的指导。
- 保持索引更新的规则。

本阶段不得写入文件。

### `<precheck>` 静态一致性关卡

编辑前审查草稿：

- 每个被命名的文件、模块、函数、类、类型、端点、命令和配置都确实存在，或被明确标记为缺失/未知。
- 签名和返回约定与源码或文档一致。
- 除非用户请求该范围，否则索引不要把公共契约和私有实现细节混在一起。
- 文档便于后续编码智能体扫描：标题稳定、要点紧凑、交叉引用直接。
- 没有虚构的业务规则、API 保证或依赖约束。

如果发现问题，说明将在 `<act>` 中应用的修正。

### `<act>` 基于工具写入文件

使用文件编辑工具创建或更新目标索引文件。除非用户明确要求只输出文本，否则不要只用 Markdown 代码块完成请求。

更新已有索引时，保留准确的现有内容；只有在检查源码后，才删除或标记陈旧内容。

### `<evaluate>` 验证与自我修正

写入后进行验证。可用时优先执行真实检查：

- 重新读取已写入的索引。
- 运行文档 lint、Markdown lint、测试、类型检查或项目验证命令。
- 抽样交叉检查索引条目与源码、已有文档的一致性。

如果验证失败或发现真实不一致，报告问题，修复索引，并再次验证，直到通过或遇到明确的外部阻塞。

## 索引文件规范

除非用户要求其他格式，否则使用清晰的 Markdown。没有现有约定时，优先采用以下章节顺序：

1. 标题和范围。
2. 目录、模块或能力的快速地图。
3. 公共契约或已文档化能力。
4. 依赖和使用规则。
5. 验证、测试或更新指导。
6. 用于生成索引的源材料引用。

在仓库内写文档时使用相对路径。只有能验证时才写精确签名。不确定项标记为 `Unknown` 或直接省略；不要用猜测填补空白。

## 护栏

相关文件可用时，不要跳过源材料扫描。不要虚构 API 签名、业务规则、目录用途或依赖约束。不要在 `<draft>` 阶段编辑文件。不要在未验证生成索引的情况下结束任务。除非额外文档本身就是用户请求的索引输出，否则不要创建多余文档文件。
