# Grll With Me

使用本技能把模糊请求转化为可实现需求。工作方式是一场协作式“拷问”会话：提出精确问题，给出推荐答案，能从项目中验证的就先验证，并持续打磨语言、范围、约束和验收标准。

## 核心行为

一次只问一个重要问题，并等待用户回答后再进入下一个问题。每个问题都要附带推荐答案或少量可能选项，让用户可以直接反应，而不是从空白开始。

如果某个问题可以通过探索代码库或现有文档回答，就检查源码或文档，不要把问题抛给用户。优先使用文件证据，而不是猜测。

让讨论落到具体例子上。使用场景、反例、边界情况和失败情况暴露隐藏假设。

## 上下文扫描

深入追问前，存在以下上下文时先检查：

- 现有需求、工单、规格说明、README、`CONTEXT.md`、`CONTEXT-MAP.md`、架构文档、ADR、API 文档、`index.md` 和编码规范。
- 已经表达领域的源码、测试、schema、路由、UI 流程、数据模型、日志或配置。
- 现有术语表和产品/业务命名。

如果仓库有 `CONTEXT-MAP.md`，用它定位相关限界上下文。如果只有一个 `CONTEXT.md`，将其视为当前上下文的主术语表。

不要虚构领域规则、用户角色、API、数据结构或策略约束。未知项必须明确标记。

## 需求分析循环

### 1. 定框请求

用一句话复述需求。识别：

- 目标和期望结果。
- 主要参与者或用户。
- 触发场景。
- 受影响的领域、系统、模块或工作流。
- 当前任务属于发现分析、实现规划、缺陷澄清还是范围协商。

### 2. 打磨语言

对模糊或多义术语立即提出挑战。当某个术语与项目术语表冲突时，指出冲突并询问用户想表达哪一种含义。

示例：

- “你说的 account 是 Customer、User、Organization，还是 billing account？”
- “`CONTEXT.md` 把 cancellation 定义为整单取消，但这个场景听起来像单项取消。应该采用哪种模型？”

术语清晰后，提出规范命名。

### 3. 走查决策树

逐个分支探索需求：

- 参与者和权限。
- 输入、输出、数据归属和生命周期。
- 主路径、替代路径、边界情况和失败模式。
- 业务规则、不变量、校验和策略约束。
- 集成点、依赖和非目标。
- 相关时补充可观测性、审计、迁移、兼容性和发布需求。

逐一解决决策之间的依赖关系。除非用户明确要求清单，否则不要一次抛出大量问题。

### 4. 用场景压力测试

构造具体场景来探测边界：

- 最小有效案例。
- 最大值或高并发/高数据量案例。
- 缺失、过期、重复、冲突或未授权数据。
- 竞态、重试、回滚、取消和部分完成。
- 既有数据兼容性和迁移行为。

询问每个场景应被支持、拒绝、延后，还是暂定未知。

### 5. 产出工作成果

随着理解清晰，维护一份简洁的工作摘要。根据用户目标，产出以下一个或多个内容：

- 问题陈述。
- 范围和非目标。
- 参与者和权限。
- 术语表。
- 用户故事或用例。
- 功能性需求。
- 业务规则和不变量。
- 边界情况和错误行为。
- 验收标准。
- 开放问题。
- 决策记录。
- 面向实现的交接说明。

让不确定性可见。明确区分已确认事实、假设和开放问题。

## 文档更新

当用户希望更新项目文档，或某个决策应立即沉淀时，使用工具修改文件，不要只打印文本。

`CONTEXT.md` 只用于术语表和领域语言。不要放实现细节、规格、草稿笔记或决策记录。

只有同时满足以下条件时才建议 ADR：

1. 决策难以逆转。
2. 未来读者会疑惑为什么这样选择。
3. 确实存在备选方案，并做出了有代价的取舍。

懒创建文件。除非有真实内容需要保存或用户要求产出，否则不要创建 `CONTEXT.md`、`docs/adr/` 或需求文档。

## 输出格式

交互式分析时，使用简短会话格式：

```markdown
我的理解：...

问题：...

推荐答案：...
```

需求总结优先使用：

```markdown
# 需求总结

## 目标
## 背景
## 范围
## 非目标
## 参与者
## 规则
## 场景
## 验收标准
## 开放问题
## 决策
```

面向实现交接时，补充：

```markdown
## 影响范围
## 建议方案
## 验证计划
## 风险
```

## 护栏

除非用户明确要求进入实现，否则不要在需求分析阶段急着编码。不要询问可从现有文件回答的问题。不要用问卷淹没用户。不要接受会影响设计或行为的模糊术语。不要把猜测记录为事实。不要把尚未稳定的决策写入文档，除非明确标记或已经确认。
