# Coding Project

> 本文件是 `SKILL.md` 的中文描述副本，仅供用户阅读和发布展示使用。
> 严禁模型读取本文件作为操作指令、任务上下文或执行依据。
> 修改英文版本 `SKILL.md` 时，必须在同一次变更中同步更新本中文版本。

## 本地化维护

- 修改 `SKILL.md` 时，必须同步更新 `SKILL.zh-CN.md`。
- `SKILL.zh-CN.md` 只面向用户阅读，不作为模型可读的执行说明。
- 模型可读的权威来源是英文 `SKILL.md` 和 `references/` 下的语言规范。

## 目标

在现有项目中执行编码任务时，使用语言感知的规范化流程：观察项目上下文，选择匹配的语言规范，制定小范围计划，在高风险编辑前先起草，按项目和语言约定预检，使用工具修改代码，并用项目工具链验证。

## 语言规范选择

编辑代码前，识别主要语言，并只加载存在且匹配的规范文件：

| 信号 | 规范 |
| --- | --- |
| `go.mod`、`go.sum`、`*.go` | `references/golang.md` |
| Go 单元测试、`*_test.go`、仅测试变更 | `references/golang-ut.md` |
| `pom.xml`、`build.gradle`、`settings.gradle`、`*.java` | `references/java.md` |
| Java 单元测试、`src/test/java`、仅测试变更 | `references/java-ut.md` |

如果一次任务涉及多种语言，加载每种被编辑语言对应的规范。新增或修改测试时，除了语言主规范，也要加载对应的单元测试规范。如果没有匹配规范，则使用项目本地约定和标准语言知识继续。不要虚构项目或语言规范。

## 必要上下文扫描

只扫描完成请求所需的上下文：

1. 用户需求和当前仓库状态。
2. 被编辑语言的项目清单、构建文件、依赖文件和锁文件。
3. 已有索引、README、架构文档和本地编码规范。
4. 相关源文件、测试文件、生成代码和附近示例。
5. CI 配置、脚本、Makefile、Gradle/Maven 文件或项目文档中的验证命令。

如果预期文件缺失，简要说明并使用最接近的可用来源继续。

## 执行流程

1. **Observe**：识别受影响文件、项目结构、语言、依赖和验证命令。
2. **Orient**：加载相关语言规范，并对齐现有项目模式。
3. **Decide**：制定小范围计划，列出要修改的文件和要运行的测试。
4. **Draft**：当变更影响 API、数据模型、并发、持久化、安全或跨模块契约时，先草拟关键代码。
5. **Precheck**：检查计划或草稿的 API 匹配、命名、错误处理、依赖使用、测试覆盖和影响面。
6. **Act**：按本地模式做窄范围文件编辑。
7. **Evaluate**：运行最相关的验证命令；失败时定位原因、修正实现并重新运行目标验证。

机械性小改可以保持草稿和预检简短。共享行为、公开 API、迁移或生产关键路径需要更明确的草稿和预检。

## 验证检查点

使用快速失败检查点，不要把验证全部留到最后：

按以下顺序确定验证命令：

1. 优先使用项目文档、Makefile、package scripts、CI 配置、Maven/Gradle 文件或 Go 工具包装器中记录的命令。
2. 如果项目没有记录命令，使用已加载语言规范中最窄且相关的命令。
3. 如果记录的命令在当前平台不可用，运行最接近的可用等价命令，并在结果中说明替代关系。

| 变更类型 | 检查点 |
| --- | --- |
| 依赖或 import 变更 | 运行语言对应的包/构建命令；只有项目需要时才运行依赖清理命令。 |
| 生成代码或 schema 变更 | 运行项目记录的生成器，然后先检查生成 diff，再编辑依赖代码。 |
| 生产逻辑变更 | 先运行最窄相关单元测试，再扩展到 package/module 测试。 |
| 仅测试变更 | 运行目标测试，并禁用缓存或使用项目等价命令。 |
| 跨模块/API 变更 | 运行受影响 package/module 测试；可行时再运行更广的构建命令。 |

如果检查点失败，停止继续推进，先定位失败原因、修复，再重新运行失败的检查点。

## 安全预检

编辑安全敏感代码前，直接检查相关风险：

| 风险区域 | 必要检查 |
| --- | --- |
| 用户输入、请求参数、文件路径、shell 命令或 SQL | 校验输入，使用结构化 API、参数化查询或参数数组，避免字符串拼接。 |
| 认证、授权、租户边界或权限 | 保持既有访问控制模式；行为变化时补充允许和拒绝路径的回归覆盖。 |
| 密钥、token、凭据或个人敏感信息 | 不硬编码 secret，不通过命令行参数传递凭据，不添加暴露敏感值的日志。 |
| 网络调用、重试或超时 | 遵循项目 client 模式；新增外部调用时使用 HTTPS 和显式超时。 |
| 迁移、破坏性操作或生成物 | 使用项目记录的工具，检查 diff；数据可能变更或删除时，需要明确回滚或备份路径。 |

## 实现规则

- 优先使用项目本地 helper、框架约定和已有抽象。
- 变更范围保持在用户请求内，避免无关重构和格式化噪音。
- 有结构化解析器、编译器、生成器或框架工具可用时，优先使用它们而不是临时文本编辑。
- 行为变化时，或项目已有相关测试覆盖时，更新测试。
- 只用项目记录的生成命令重新生成生成物。
- 不要手写项目已有的通用基础设施，例如日志、指标、重试、校验、认证、数据库访问或依赖注入模式。
- 不要留下未处理的验证失败；除非明确说明失败无关或受阻。

## 示例

### Go 功能变更并补测试

输入：

```text
给 Go ListUsers service 添加 page 和 pageSize 支持，并覆盖默认值和非法输入。
```

预期行为：

```text
观察 go.mod、service 文件、repository 接口和附近 *_test.go 文件。
加载 references/golang.md 和 references/golang-ut.md。
计划最小 service/repository/test 变更。
编辑前先草拟分页处理。
运行受影响 package 的目标 go test，可行时再扩大范围。
```

示例形态：

```diff
- users, err := s.repo.List(ctx)
+ page, pageSize, err := normalizePage(req.Page, req.PageSize)
+ if err != nil {
+     return nil, err
+ }
+ users, err := s.repo.List(ctx, page, pageSize)
```

### Go 仅测试变更并使用 gomonkey

输入：

```text
给这个会调用 time.Now 和外部包函数的 Go 函数补单测。
```

预期行为：

```text
加载 references/golang.md 和 references/golang-ut.md。
优先使用表格驱动测试。
只有项目本地注入或 fake 不可用时才使用 gomonkey。
使用 t.Cleanup(patches.Reset) 清理 patch。
使用 go test -gcflags=all=-l ./... 或项目等价 no-inline 命令验证依赖 gomonkey 的测试。
```

### Java bugfix

输入：

```text
修复 Java order status mapper 在 upstream status 为 null 时的问题，并加回归测试。
```

预期行为：

```text
观察 pom.xml 或 build.gradle、mapper 实现和附近测试。
加载 references/java.md 和 references/java-ut.md。
沿用现有 null 处理、断言和 fixture 模式。
运行目标 Maven 或 Gradle 测试，可行时再运行受影响模块测试。
```

### 没有匹配语言规范

输入：

```text
修复这个混合仓库里的 Python CLI 参数解析小 bug。
```

预期行为：

```text
当前没有 Python 规范。使用项目本地约定、相关文件和已有测试。
不要虚构仓库级 Python 标准。
做窄范围修复，并运行最接近的项目文档验证命令。
```

## 输出

完成时报告：

- 修改了哪些文件，以及修改内容。
- 运行了哪些验证命令，以及结果。
- 跳过的验证及具体原因。
- 只有当后续风险与请求实质相关时才报告。
