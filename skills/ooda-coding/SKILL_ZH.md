# OODA 编码

使用本技能处理需要严谨上下文收集、架构对齐、先草稿后实现、基于工具编辑和验证闭环的编码任务。核心原则是：观察项目，面向标准和索引定位，决定小范围计划，编辑前先草拟，像审查者一样预检，用工具执行修改，并持续评估直到结果正确。

## 必需上下文扫描

在设计或编辑代码前，扫描目标范围内存在的项目上下文：

1. 用户需求和当前仓库状态。
2. 现有索引文件，例如 `index.md`、模块索引、API 索引或基础设施索引。
3. 项目 README、架构文档、包清单、路由/API 文档、schema、测试约定和编码规范。
4. 存在业务文档时，特别读取 `业务层/README.md`、`业务层/编码规范.md` 和 `基础设施层/index.md`。
5. 与请求变更直接相关的源码文件和测试。

如果预期文件缺失，要说明缺失并继续使用最接近的可用源材料。不要虚构 API、业务规则、编码规范或模块边界。

## 输出契约

对于实质性编码工作，可见推理和结果总结必须按以下顺序组织：

```xml
<observe>
</observe>
<orient>
</orient>
<decide>
</decide>
<draft>
</draft>
<precheck>
</precheck>
<act>
</act>
<evaluate>
</evaluate>
```

用户可见文本保持简洁。工具调用发生在可见标签之外，但工作总结必须遵循该顺序。

## OODA-E 工作流

### `<observe>` 需求与项目侦察

用一句话总结请求的代码变更。识别受影响的模块、层级、功能、API 或业务链路。

读取相关文档、索引、源码、测试和配置。当存在 `index.md` 或基础设施索引时，先用它定位公共契约和可复用能力，再阅读实现细节。

对于业务仓库，从业务文档中提取影响实现的隐含规则，并引用相关的现有基础设施 API、函数签名、输入要求、返回约定和使用约束。

### `<orient>` 标准、契约与依赖对齐

将预期实现与已发现的项目规则对齐：

- 命名、格式、错误处理、日志、依赖注入、模块边界和分层。
- 现有公共契约、类型、schema、路由模式、持久化模式和测试风格。
- 索引中已经文档化的基础设施能力。

内部检查当前变更是否准备手写某种本应来自现有代码或基础设施的通用能力，例如网络请求、缓存、时间处理、日志、配置、重试、序列化、认证、持久化、校验、队列或存储。如果是，切换到已文档化的现有能力。

优先进行保持当前架构的小范围修改。除非安全完成用户请求确实需要，否则避免无关重构。

### `<decide>` 实现计划

列出具体实现步骤：

- 目标文件。
- 需要修改的 imports、函数、类、类型、路由、schema 或测试。
- 核心逻辑以及错误/边界情况处理。
- 将运行的验证命令或静态检查。

计划必须限制在用户请求的行为和已发现架构范围内。

### `<draft>` 脑内沙盒草稿

在修改文件前，先用文本草拟预期代码或补丁内容。这是静态沙盒阶段；本阶段不得调用文件写入工具。

草稿必须完整到足以检查：

- imports 和依赖。
- 函数签名和类型流。
- 控制流、异步行为和错误处理。
- 基础设施/API 调用及参数兼容性。
- 测试用例或验证策略。

对于非常小的改动，紧凑的补丁草图即可，但仍必须先草拟再编辑。

### `<precheck>` 静态质量关卡

切换到严格审查者视角，在编辑前检查草稿：

- 依赖健康：所有引用的函数、变量、类型、文件和基础设施 API 都存在，或是计划新增项。
- 类型与契约健康：参数、返回值、schema 和错误约定与源码和索引一致。
- 语法健康：不存在明显的语法、作用域、异步、导入、模块格式或框架生命周期错误。
- 标准健康：命名、日志、分层、错误处理和测试符合项目约定。
- 影响范围健康：没有无关行为变化、隐藏全局状态、脆弱 mock 或可避免的兼容性风险。

如果发现问题，说明问题以及将在 `<act>` 中如何修正。

### `<act>` 基于工具写入文件

使用可用文件编辑工具把修正后的变更应用到真实文件。除非用户明确要求只输出文本，否则不要只用 Markdown 代码块完成本阶段。

尊重现有工作区。不要回滚用户变更。将编辑限制在目标范围内。

### `<evaluate>` 动态验证与自我修正

编辑后验证结果。可用时优先执行真实命令：

- 单元测试、集成测试、目标测试文件或快照测试。
- 类型检查、lint、格式检查、编译、构建或项目特定验证。
- 对小改动或命令受阻的场景，重新读取修改后的文件并做最终静态审查。

如果验证失败或发现真实缺陷，报告失败的检查和相关错误，然后不要等待用户，继续自我修正循环。回到合适的早期阶段，修复问题，并重新验证，直到检查通过或遇到明确外部阻塞。

## 索引意识

当存在 `index.md` 或 OODA 风格索引时，将其视为导航层和契约层：

- 在打开私有实现文件前，先用它寻找公共 API 和预期用法。
- 保持实现与已文档化契约一致。
- 如果代码变更有意更新公共契约，并且这在用户请求范围内或为了保持一致性是必要的，则同步更新相关索引或文档。
- 如果索引看起来陈旧，先与源码核对并说明不一致，再决定是否依赖它。

## 护栏

相关文件存在时，不要跳过上下文扫描。不要虚构缺失的业务规则、API 签名或编码规范。不要手写已有基础设施能力。不要在 `<draft>` 阶段编辑文件。验证失败后不要结束任务，除非阻塞来自明确的外部因素。不要用无关重构或文档噪音扩大任务范围。
