# Exploring Project

> 本文件是 `SKILL.md` 的中文描述副本，仅供人工阅读和发布展示使用。
> 严禁模型读取本文件作为操作指令或任务上下文。
> 修改英文版本 `SKILL.md` 时，必须在同一次变更中同步更新本中文版本。

## 目标

在不通读整个仓库的前提下，建立足够的项目理解，以便回答问题或修改指定区域；使用 CodeMap 和项目指南缩小范围，然后阅读必要的源文件来验证行为。

## 澄清门

在深入探索之前，先复述用户期望的结果，并识别目标行为、功能或模块。

只有当歧义会改变查找位置或修改内容时，才向用户提出澄清问题。问题要少而具体，通常为 1-3 个。每个问题都必须提供你的推荐答案，方便用户快速接受或纠正。如果请求已经足够明确，就基于合理假设继续，并简要说明这些假设。

优先询问用户可见行为、受影响的功能/模块、复现步骤或示例，以及框架、API、测试或兼容性等约束。

不要进行宽泛的需求访谈。只提出选择探索路径或实现范围所必需的澄清问题；其他不确定性应通过有针对性的项目探索来解决。

## CodeMap 优先探索

当当前项目可用 `codemap` MCP server 时，在打开源文件之前，先使用它映射可能相关的链路、函数、模块、路由、流程和风险区域。

1. 先用用户需求或澄清后的需求摘要调用 `find_change_points`。除非目标很宽泛，否则使用较小的 `top_k`，例如 5。
2. 如果请求面向功能，调用 `get_feature_map` 或 `get_navigation_hints` 来识别入口文件和相关模块。
3. 如果目标点名了路由、模块、流程或函数，使用聚焦工具：`search_route`、`search_module`、`related_modules`、`search_flow`、`call_graph` 或 `impact_analysis`。
4. 如果 CodeMap 不可用、索引过期、不支持当前项目或结果太模糊，则回退到轻量仓库发现：`rg`、`rg --files`、清单文件、路由、导入和测试。

| 用户请求中的目标 | CodeMap 工具 | 通过阅读以下内容验证 |
| --- | --- | --- |
| 宽泛需求或修改请求 | `find_change_points` | 候选文件、函数、路由和测试 |
| 功能或用户工作流 | `get_feature_map`、`get_navigation_hints` | 入口文件、相关模块和测试 |
| HTTP/API 路由 | `search_route` | 处理器、服务/用例和路由测试 |
| 模块或包 | `search_module`、`related_modules` | 模块实现、导入和依赖方 |
| 流程或调用链 | `search_flow`、`call_graph` | 调用方/被调用方实现和集成测试 |
| 函数影响面 | `impact_analysis` | 调用方、行为契约和受影响测试 |

将 CodeMap 输出视为导航线索，而不是事实结论。解释行为或编辑代码前，必须阅读被引用的实现或测试来验证关键判断。

## 生成索引桥接

当 `.agent/PROJECT_INDEX.md`、`.agent/NAVIGATION.md`、`.agent/CHANGE_GUIDE.md` 或 `.agent/FEATURES.md` 存在时，在宽泛发现之前先把它们作为范围过滤器。

1. 先选择最小可用指南：业务能力读 `.agent/FEATURES.md`，起始文件和相关模块读 `.agent/NAVIGATION.md`，变更触点读 `.agent/CHANGE_GUIDE.md`，整体系统上下文读 `.agent/PROJECT_INDEX.md`。
2. 只阅读与用户请求的功能、模块、路由、流程或变更类型相关的小节。
3. 如果存在，提取 `Feature`、`Start From`、`Related Modules`、`Related Routes`、`Related Flows`、`Tests`、`Risk` 和 `Source Evidence` 字段作为候选线索。
4. 将提取出的名称转换成聚焦 CodeMap 调用：模块 -> `search_module` 或 `related_modules`；路由 -> `search_route`；流程 -> `search_flow` 或 `call_graph`；宽泛变更 -> `find_change_points`。
5. 如果 `.agent/CHANGE_GUIDE.md` 匹配请求的变更类型，使用其中的 `Touch`、`Typical Flow`、`Tests` 和 `Risk` 条目选择第一批需要验证的文件。
6. 如果索引字段缺失、过期，或与 CodeMap/源码文件冲突，明确说明，并以已验证源码为准，而不是以生成文档为准。

## 核心流程

1. 优先从已有项目指南开始：`README`、`.agent/PROJECT_INDEX.md`、`.agent/NAVIGATION.md`、`.agent/CHANGE_GUIDE.md`、`.agent/FEATURES.md`、文档、架构说明或贡献指南。
2. 如果请求的歧义足以影响探索或实现路径，则应用澄清门。
3. 当 `.agent/` 索引文件存在时，使用生成索引桥接。
4. 如果可用，先运行 CodeMap 优先探索，在阅读源代码前识别可能相关的链路、函数和文件。
5. 只有当项目指南和 CodeMap 无法提供足够范围时，才使用轻量命令识别项目形态：文件树、清单文件、包配置、路由、测试和主要入口点。
6. 在打开大量文件之前，先用业务术语命名目标功能、模块或行为。
7. 沿引用关系从入口追踪到实现：路由/命令/UI -> 服务/用例 -> 模型/存储/集成 -> 测试。
8. 只打开与目标区域相关的文件。除非第一轮探索失败，否则避免全仓库扫描。
9. 将发现总结为可执行上下文：相关文件、职责边界、数据/控制流、风险和下一步编辑位置。

## 检查点

- 在阅读项目指南和完成 CodeMap 优先探索后，先列出候选文件、函数、路由和不确定点，再打开大量文件。
- 在解释行为或编辑代码前，通过阅读被引用的实现或测试来验证相关 CodeMap 线索。
- 在结束探索前，报告已验证的流程和下一步编辑位置。如果验证失败，说明未找到什么，并继续进行有针对性的发现。

## 调查规则

- 优先使用 `rg` 和 `rg --files` 做发现。
- 当 CodeMap 可用且相关时，优先用它完成第一轮探索，而不是直接阅读源文件。
- 优先使用已有索引和生成文档，而不是重新发现所有内容。
- 将生成索引文件作为导航线索，而不是事实结论；通过 CodeMap、源码文件或测试验证关键路径和行为。
- 使用依赖边、导入、路由注册、测试名称和配置文件来缩小搜索范围。
- 将代码中的命名视为证据，而不是结论；需要在实现或测试中验证行为。
- 明确表达不确定性。用 `unknown` 或 `not found`，不要猜测。
- 一旦下一步行动足够明确，就停止探索。

## 输出形式

报告探索结果时保持简洁：

```text
项目形态：
- ...

相关文件：
- path: 为什么重要

CodeMap 线索：
- 工具结果 -> 已验证/未验证

流程：
- 入口 -> 实现 -> 持久化/集成

风险：
- ...

下一步修改位置：
- ...
```

对于代码修改，应使用这些发现来保持改动范围最小，并避免无关重构。

## 示例

### 明确的修改请求

输入：

```text
修复订单取消后没有返还库存的问题。
```

预期行为：

```text
复述目标。针对需求调用 `find_change_points`，然后对任何被点名的路由、流程、模块或函数使用聚焦 CodeMap 工具。只阅读验证取消 -> 库存流程所必需的候选实现和测试，再提出或执行修改。
```

### 模糊请求

输入：

```text
优化登录流程。
```

预期行为：

```text
提出 1-3 个澄清问题，因为“优化”可能改变探索路径。每个问题都提供推荐答案，例如：“推荐：聚焦密码登录失败后的用户可见错误。”用户接受或纠正推荐后，先运行 CodeMap，再用源码/测试验证。
```

### CodeMap 不可用或结果较弱

输入：

```text
找出这个项目在哪里生成发票。
```

预期行为：

```text
先尝试 CodeMap。如果不可用、索引过期、不支持当前项目或结果太模糊，则阅读 `README`、`.agent/PROJECT_INDEX.md`、`.agent/NAVIGATION.md`、`.agent/CHANGE_GUIDE.md`、`.agent/FEATURES.md`、项目文档、清单文件、路由和测试，再用 `rg`/`rg --files` 定位发票入口点。最终答案必须通过阅读相关实现或测试来验证。
```
